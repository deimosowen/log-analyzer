# Log Analyzer Roadmap

Документ фиксирует направление развития проекта, чтобы реализация шла последовательно и не расползалась в случайные улучшения.

## Принципы развития

- Сначала усиливаем основной сценарий: загрузил логи, выбрал проблему, увидел связанные события, сформировал отчет.
- Каждая новая функция должна помогать быстрее понять инцидент, а не просто добавлять еще один экран.
- UI остается рабочим инструментом: компактный, предсказуемый, без перегруза и маркетинговой декоративности.
- Большие данные обрабатываются фоново, с понятным прогрессом и ограничениями.
- Архитектура не должна завязывать доменную логику на Blazor-компоненты или конкретную БД.

## Правило Покрытия БД

- PostgreSQL используется только как production metadata storage: пользователи, проекты, загрузки, файлы, ошибки импорта, права, заметки, настройки, audit log.
- SQLite остается local/dev fallback для metadata и легким event-store режимом разработки.
- ClickHouse остается основным production event-store для больших логов, timeline и агрегатов событий.
- Любая новая metadata-сущность должна иметь миграции и repository-поддержку для PostgreSQL и SQLite.
- Любой новый event/read-side запрос должен иметь реализацию для ClickHouse и SQLite. PostgreSQL event-store не добавляем, пока явно не изменим архитектурное решение.
- Тесты должны покрывать общую application-логику и минимум один storage backend; для metadata-задач обязательно проверять PostgreSQL migration catalog и SQLite fallback.

## Этап 0. Переезд Metadata DB С SQLite На PostgreSQL

Цель: до активного роста функциональности заменить SQLite как основное хранилище metadata на PostgreSQL.

### Почему Это Важно Сейчас

SQLite удобен для MVP и локального запуска, но дальше проекту понадобятся командная работа, роли, шаринг, audit log, retention, админка и больше конкурентных операций. PostgreSQL лучше подходит для серверного multi-user сценария: транзакции, блокировки, индексы, миграции, эксплуатация и бэкапы будут предсказуемее.

### Пользовательская ценность

Пользователь получает более устойчивую серверную установку: данные проектов, пользователи, загрузки, комментарии и права доступа живут в полноценной БД, которую проще бэкапить, переносить и обслуживать.

### Функции

- Добавить PostgreSQL как основную metadata DB.
- Оставить SQLite только как optional local/dev режим, если это не усложнит архитектуру.
- Перенести таблицы metadata:
  - users;
  - projects;
  - upload sessions;
  - log files;
  - import errors;
  - schema migrations.
- Добавить PostgreSQL migrator с миграциями по одному файлу на версию.
- Добавить PostgreSQL metadata repository.
- Обновить Docker Compose:
  - сервис `postgres`;
  - bind mount для `/var/lib/postgresql/data`;
  - env-настройки подключения.
- Обновить README и `.env.example`.
- Подготовить стратегию миграции существующих данных из SQLite в PostgreSQL.

### Критерии готовности

- Новая установка поднимается с PostgreSQL без SQLite.
- Все текущие сценарии работают:
  - авторизация;
  - создание проекта;
  - загрузка логов;
  - индексация;
  - анализ инцидента;
  - отчет.
- Миграции PostgreSQL применяются автоматически при старте.
- Docker Compose хранит данные PostgreSQL в папке рядом с compose или в явно заданном volume.
- Тесты покрывают PostgreSQL migration catalog и базовые repository-сценарии.
- README содержит инструкции запуска, backup и restore PostgreSQL.

### Миграция Данных

Если к моменту переезда уже есть production-данные в SQLite, нужен отдельный migration tool или команда:

- читает текущую SQLite metadata DB;
- переносит пользователей, проекты, upload sessions, log files и import errors;
- сохраняет идентификаторы;
- проверяет количество записей до и после;
- не переносит indexed events из ClickHouse, потому что они уже живут отдельно.

Для первого production-переезда можно сделать ручной one-shot console command, но его нужно держать в репозитории и документировать.

### Backup И Restore

Минимальные команды для документации:

- `pg_dump` для backup metadata;
- `psql` для restore;
- отдельный backup ClickHouse;
- отдельный backup storage folder.

### Риски

- Усложнение локального запуска.
- Нужно внимательно сохранить compatibility с текущими migration abstractions.
- Нужно не смешать metadata storage и event storage: PostgreSQL заменяет SQLite metadata, но ClickHouse остается основным event store для больших логов, а SQLite event store остается dev fallback.
- При неверных volume/bind настройках можно потерять данные, поэтому compose должен быть максимально явным.

### Рекомендуемое Решение

Сделать PostgreSQL основным metadata provider для Docker/production, а SQLite оставить только для локальной разработки на первое время.

В конфигурации использовать явный provider:

```json
"Metadata": {
  "Provider": "PostgreSQL"
}
```

Сервис-регистрация должна выбирать реализацию `IMetadataRepository` и metadata migrator по provider, а application/UI не должны знать, какая БД используется.

## Этап 1. Timeline И Карта Инцидента

Цель: дать визуальное понимание того, как развивался инцидент во времени.

### Пользовательская ценность

Пользователь видит не просто таблицу событий, а временную картину: когда начались ошибки, где был пик, какие логи участвовали, что происходило до и после выбранного события.

### Функции

- Общий timeline по выбранному проекту.
- Timeline вокруг выбранного события в анализе инцидента.
- Цветовая кодировка по уровню: fatal, error, warn, info.
- Группировка точек по временным бакетам: 1 сек, 10 сек, 1 мин, 5 мин.
- Быстрый переход из точки timeline к событиям в таблице.
- Отдельное отображение выбранного события как якоря расследования.

### Критерии готовности

- Timeline корректно работает на проектах с большим количеством IIS-событий.
- Можно выбрать временной диапазон на графике и применить его как фильтр.
- Выбранное событие визуально выделяется.
- График не ломает мобильную и desktop-компоновку.

### Технические заметки

- Добавить read model для агрегированных точек timeline.
- Для ClickHouse лучше считать бакеты на стороне БД.
- Для SQLite нужен fallback-запрос с группировкой по bucket, чтобы локальный режим не отставал от ClickHouse.
- Для PostgreSQL на этом этапе проверить только metadata-provider совместимость: проекты, log files и права доступа берутся из PostgreSQL, но события и timeline читаются из ClickHouse/SQLite event store.
- UI лучше делать отдельным компонентом, не встраивать логику в `Analysis.razor`.

### Функции

- [x] Отдельная вкладка или режим "HTTP".
- [x] Топ URL по количеству 500.
- [x] Топ URL по количеству 400.
- [x] Топ медленных запросов по `time-taken`.
- [x] Фильтры по:
  - `sc-status`;
  - `cs-uri-stem`;
  - `cs-method`;
  - `c-ip`;
  - `cs-username`;
  - `time-taken`.
- [x] Автоматический список IIS-запросов рядом с выбранной ошибкой приложения.

### Критерии готовности

- IIS-события не перегружают основной анализ.
- При окне 1 минута пользователь видит агрегаты, а не тысячи строк без структуры.
- Можно быстро раскрыть агрегат и увидеть конкретные запросы.

### Технические заметки

- [x] UI использует агрегированное представление: endpoint, status group, count, p95 time, max time.
- [x] Добавлены event-store миграции SQLite и ClickHouse.
- [x] Добавлены тесты для parser, migration catalog и SQLite IIS aggregation.

## Этап 3. Закладки, Комментарии И Вердикт

Цель: превратить анализ из просмотра данных в оформленное расследование.

### Пользовательская ценность

Пользователь может отметить важные события, оставить комментарии и зафиксировать итог: причина, влияние, решение.

### Функции

- Закладка события.
- Комментарий к событию.
- Комментарий к группе событий.
- Карточка вердикта проекта:
  - причина;
  - влияние;
  - решение;
  - статус расследования.
- Статусы проекта:
  - новый;
  - в работе;
  - разобран;
  - архив.

### Критерии готовности

- Закладки отображаются в анализе и отчете.
- Комментарии сохраняются и доступны после перезапуска.
- Вердикт можно редактировать без потери истории проекта.

### Технические заметки

- Нужны миграции metadata DB:
  - `incident_notes`;
  - `event_bookmarks`;
  - возможно `project_status`.
- Эти metadata-миграции должны быть реализованы отдельно для PostgreSQL и SQLite.
- Не хранить комментарии в event store, это metadata.
- Добавить application-сервис для работы с заметками.

## Этап 4. Улучшенный Отчет

Цель: отчет должен быть пригоден для передачи команде без ручного дописывания.

### Пользовательская ценность

После анализа пользователь получает понятный документ: что произошло, когда началось, какие логи это подтверждают, какие события важны.

### Функции

- Расширенный Markdown-отчет.
- HTML-отчет для просмотра в браузере.
- Позже: PDF-экспорт.
- Секции отчета:
  - краткое резюме;
  - выбранное проблемное событие;
  - временное окно;
  - timeline;
  - связанные события;
  - топ групп ошибок;
  - IIS summary;
  - закладки и комментарии;
  - вердикт.

### Критерии готовности

- Отчет можно сформировать из анализа инцидента одним действием.
- В отчете есть ссылки или идентификаторы событий, чтобы вернуться к деталям.
- Markdown читаем без приложения.

### Технические заметки

- Сохранить текущий `IncidentMarkdownReportBuilder`, но расширить модель запроса.
- Не смешивать построение отчета с UI.
- Если отчеты, шаблоны или история выгрузок будут сохраняться, это metadata: нужны PostgreSQL и SQLite миграции.
- Для HTML-версии можно использовать простой server-side renderer или шаблоны.

## Этап 5. Шаринг И Командная Работа

Цель: проект должен работать не только для одного пользователя, но и для команды.

### Пользовательская ценность

Инцидент может расследовать несколько человек. Владелец проекта делится доступом, участники смотрят анализ, оставляют комментарии и формируют отчет.

### Функции

- Роли:
  - владелец;
  - участник;
  - только чтение.
- Доступ к проекту по пользователям.
- Доступ к проекту по домену организации.
- Страница участников проекта.
- Audit log основных действий:
  - создан проект;
  - загружены логи;
  - добавлен комментарий;
  - сформирован отчет;
  - изменен доступ.

### Критерии готовности

- Пользователь не видит чужие проекты без доступа.
- Владелец может выдать и отозвать доступ.
- API и UI используют одинаковую модель прав.

### Технические заметки

- Вынести проверку доступа в отдельный application/security service.
- Добавить таблицы:
  - `project_members`;
  - `audit_log`.
- Таблицы доступа и audit log являются metadata: обязательны миграции и repository-поддержка для PostgreSQL и SQLite.
- Не размазывать проверки доступа по endpoint-ам.

## Этап 6. Health, Admin И Retention

Цель: сделать эксплуатацию предсказуемой.

### Пользовательская ценность

Администратор понимает состояние сервиса, объем данных и может управлять лимитами.

### Функции

- Health page:
  - PostgreSQL metadata доступен;
  - SQLite metadata/event fallback доступен в local/dev режиме;
  - ClickHouse event store доступен;
  - storage доступен;
  - очередь импорта работает.
- Страница системной статистики:
  - количество проектов;
  - количество логов;
  - размер storage;
  - количество событий;
  - размер PostgreSQL metadata;
  - размер ClickHouse.
- Настройки лимитов:
  - размер файла;
  - размер распаковки;
  - число файлов в архиве;
  - retention.
- Retention policy:
  - удалять проекты старше N дней;
  - архивировать старые проекты;
  - удалять загруженные файлы после индексации, если разрешено.

### Критерии готовности

- Админ видит, почему сервис не работает, без входа в контейнер.
- Лимиты видны в UI и совпадают с backend-настройками.
- Старые данные можно удалить управляемо, без ручной чистки volumes.

### Технические заметки

- Нужна роль администратора.
- Роль администратора, лимиты и retention-настройки хранятся в metadata: PostgreSQL и SQLite миграции обязательны.
- Для storage size нужен filesystem scanner с ограничением частоты.
- Retention лучше делать background job-ом.

## Этап 7. Производительность И Большие Загрузки

Цель: уверенно работать с большими архивами и шумными IIS-логами.

### Пользовательская ценность

Большие логи не подвешивают интерфейс, прогресс понятен, ошибки импорта видны, повторная загрузка одинаковых файлов не тратит время.

### Функции

- Дедупликация файлов по hash.
- Более подробный прогресс импорта:
  - текущий файл;
  - обработанные строки;
  - скорость обработки;
  - примерное оставшееся время.
- Отмена импорта с корректным статусом.
- Пакетная вставка событий с настраиваемым batch size.
- Настраиваемый retention для raw-файлов.

### Критерии готовности

- Большой IIS-архив не делает UI непригодным.
- Повторная загрузка файла определяется как duplicate.
- Отмена импорта не оставляет проект в неконсистентном состоянии.

### Технические заметки

- Проверить batch insert в SQLite и ClickHouse.
- Metadata-прогресс, дедупликация и статусы должны одинаково работать через PostgreSQL и SQLite metadata providers.
- Индексацию лучше держать вне UI-потока.
- Возможно, в будущем выделить worker в отдельный контейнер.

## Этап 8. Интеграции

Цель: уменьшить ручную работу по доставке логов и распространению результатов.

### Пользовательская ценность

Логи и отчеты переходят между системами автоматически: меньше ручных скачиваний, загрузок и копирования.

### Функции

- Импорт из S3/MinIO.
- Импорт из Loki.
- Импорт из Elasticsearch/OpenSearch.
- Импорт из ClickHouse таблиц.
- Webhook после завершения индексации.
- Отправка отчета в Slack, Teams или Telegram.
- Создание задачи в GitHub, Jira или YouTrack.

### Критерии готовности

- Интеграции необязательны и не усложняют локальный запуск.
- Секреты не хранятся в БД в открытом виде.
- Ошибки интеграций видны пользователю.

### Технические заметки

- Нужен слой connector-ов.
- Конфигурация интеграций должна жить отдельно от core анализа.
- Для секретов понадобится защищенное хранение или внешний secret manager; если часть конфигурации хранится в metadata DB, нужны PostgreSQL и SQLite миграции.

## Этап 9. AI-Assisted Анализ

Цель: добавить помощника, который объясняет найденный контекст, но не заменяет факты из логов.

### Пользовательская ценность

Пользователь быстрее получает первичную гипотезу: что произошло, какие события связаны, что проверить дальше.

### Функции

- Суммаризация выбранного временного окна.
- Объяснение stack trace простым языком.
- Поиск похожих прошлых инцидентов.
- Черновик postmortem.
- Предложение возможного root cause.

### Критерии готовности

- AI-вывод всегда ссылается на конкретные события.
- Пользователь видит, что это гипотеза, а не факт.
- Можно отключить AI полностью.

### Технические заметки

- Не отправлять секретные логи наружу без явного разрешения.
- Нужна настройка redaction для токенов, паролей, персональных данных.
- AI должен работать поверх уже выбранного контекста, а не читать всю БД.
